# 체크리스트 관리 기능 요구사항 명세서

> 상태: 작성 중 (주요 정책 확정, 일부 TBD 항목 존재 — [16. 미확정 항목](#16-미확정-항목tbd) 참고)
> 대상 서비스: 현풍닭칼국수 매장 운영 체크리스트 (신칼)

## 1. 문서 목적

본 문서는 운영 체크리스트, 식자재 체크리스트, 공산품 체크리스트의 화면 구성, 데이터 구조, 입력 항목, 저장/동시성 정책, 이력 관리 요구사항을 정리한다. 기술 스택 결정은 [ADR 0001](../adr/0001-api-db-stack.md)을 따른다.

## 2. 용어 정의

| 용어 | 정의 |
|---|---|
| 영업일(business_date) | 체크리스트 기록의 기준 날짜. KST 기준 캘린더 날짜(자정 기준)를 그대로 사용한다. 새벽 영업이 없으므로 별도 영업일 보정은 하지 않는다. |
| 체크리스트 종류(checklist_type) | 운영(12종) / 식자재 / 공산품. 저장 단위의 기준이 된다. |
| 저장 단위 | 한 번의 저장 버튼으로 기록되는 묶음. (지점 + 영업일 + 체크리스트 종류) 1건. |
| 품목 마스터 | 식자재/공산품의 기준 품목 목록. 설정 페이지에서 관리한다. |
| 참조 모델 | 기록 행(line)은 품목을 `item_id`로 **참조만** 하고, 품목명·카테고리명 등 **이름(카탈로그) 값은 마스터를 JOIN해 표시**하는 방식. 설정에서 이름을 바꾸면 과거 기록 표시도 함께 바뀐다. |
| 입력값 동결 | 그날 사용자가 입력/선택한 값(단위, 단위옵션, 기준/실제 사용량, 판정/원인, 재고, 단가)은 기록 행에 그대로 저장되어 마스터 변경의 영향을 받지 않고 과거 그대로 보존되는 것. |
| 전회차 단가 | 해당 품목의 직전 기록 공급단가. 기본은 전날 기록, 전날이 없으면 그 이전 가장 최근 기록일을 사용한다. |

## 3. 체크리스트 구성 및 저장 단위

| 체크리스트 | 목적 | 저장 단위 수 |
|---|---|---|
| 운영 체크리스트 | 매장 운영 상태 점검 (오픈/운영/마감 포지션별) | 12종 |
| 식자재 체크리스트 | 식자재 기준/실제 사용량, 단가 변동 관리 | 1종 |
| 공산품 체크리스트 | 공산품 재고 및 주간 평균 사용량 관리 | 1종 |

- **저장 버튼은 체크리스트 단위로 총 14개**(운영 12 + 식자재 1 + 공산품 1)이다. 각 체크리스트는 독립적으로 저장/조회된다.

## 4. 공통 화면 및 데이터 요구사항

| 구분 | 요구사항 |
|---|---|
| 진입 인증 | 초기 진입 화면에서 회사에서 정한 공용 비밀번호를 입력해야 진입할 수 있다. ([5. 인증/접근](#5-인증접근-정책) 참고) |
| 체크리스트 선택 | 좌측 사이드탭에서 운영(12종) / 식자재 / 공산품 체크리스트를 선택할 수 있어야 한다. |
| 지점 선택 | 화면 최상단 오른쪽 영역에 지점 선택 기능을 제공한다. |
| 지점 옵션 | 광화문, 서면 |
| 날짜 선택 | 지점 선택 영역 오른쪽에 영업일 선택 기능을 제공한다. |
| 데이터 저장 | 운영/식자재/공산품 체크리스트 모두 DB에 저장한다. |
| 이력 조회 | 지점 및 영업일별로 이전 체크리스트 기록을 조회할 수 있어야 한다. |
| 이력 수정 | 과거 영업일의 기록도 조회 후 수정/재저장이 가능해야 한다. |
| 저장된 기록 공유 | 특정 (지점·영업일·체크리스트)가 저장되어 있으면, 다른 사람이 열람할 때 저장된 데이터가 보여야 한다. |
| 기준 데이터 관리 | 식자재/공산품 품목과 식자재 카테고리는 설정 페이지에서 추가/수정/삭제할 수 있어야 한다. ([9. 설정 페이지](#9-설정-페이지-품목카테고리-마스터-관리) 참고) |
| 반응형 | 태블릿, 모바일, 데스크탑에서 모두 사용 가능하도록 반응형(responsive) 레이아웃으로 구성한다. |
| 네트워크 | 네트워크가 연결된 환경에서만 사용한다. 오프라인 저장은 범위 외이다. |

## 5. 인증/접근 정책

| 항목 | 정책 |
|---|---|
| 사용자 | 지점 관리자/직원 10명 내외가 공용으로 사용한다. |
| 인증 방식 | 개별 계정/로그인 없이, 공용 비밀번호 1개 입력으로 진입한다. |
| 권한 분리 | 지점별 접근 제한, 역할 권한은 현재 단계에서 두지 않는다. (모든 진입자가 모든 지점·체크리스트 접근 가능) |
| 사용자 식별 | 누가 기록/삭제했는지는 시스템적으로 추적하지 않는다. 대신 각 체크리스트에 담당자명을 입력해 식별한다. |
| 감사 로그 | 전용 감사 로그 테이블은 두지 않는다. (저장공간 고려, 담당자 입력으로 갈음) 레코드의 `updated_at`/`version`/담당자 필드로 마지막 저장 정보만 확인한다. |

## 6. 운영 체크리스트

### 6.1 구성

- 운영 체크리스트는 포지션·단계별로 **12종**으로 구분되며, 각 종은 독립 저장 단위이다.
- 각 종은 단계(phase: 오픈 / 운영 / 마감)와 여러 섹션을 가지며, 섹션은 다수의 점검 항목을 가진다.
- **각 점검 항목은 boolean(체크 여부)** 으로만 관리한다.

| ID | 단계 | 체크리스트명 |
|---|---|---|
| 1 | 오픈 | 매장 오픈 [홀] |
| 2 | 오픈 | 매장 오픈 [주방 서브] |
| 3 | 오픈 | 매장 오픈 [주방 메인] |
| 4 | 운영 | 매장 운영 [홀 - 메인] |
| 5 | 운영 | 매장 운영 [홀 - 서브] |
| 6 | 운영 | 매장 운영 [홀 - 서서브] |
| 7 | 운영 | 매장 운영 [주방 메인] |
| 8 | 운영 | 매장 운영 [주방 서브] |
| 9 | 운영 | 매장 운영 [주방 서서브] |
| 10 | 마감 | 매장 마감 [홀] |
| 11 | 마감 | 매장 마감 [주방 메인] |
| 12 | 마감 | 매장 마감 [주방 서브] |

> 각 체크리스트의 섹션/항목 원본 정의는 API 템플릿(`apps/api/src/operationTemplates.ts`)을 기준으로 한다. 항목 텍스트가 바뀌면 마스터로 분리해 관리하는 것을 추후 검토한다.

### 6.2 저장 항목

| 항목 | 설명 |
|---|---|
| 항목별 체크 여부 | 각 점검 항목의 boolean 값 |
| 총점수 | 체크된 항목 수(**항목당 1점**)로 자동 계산한다. 비율/가중치는 사용하지 않는다. 화면에는 `체크수 / 전체항목수`로 함께 표시할 수 있다. |
| 담당자 목록 | 담당자명 1칸 + 체크리스트 템플릿 `roles` 기준 담당 포지션 입력칸. 담당자명 포함 최대 3칸이며 JSON 배열로 저장한다. |

## 7. 식자재 체크리스트

### 7.1 컬럼 구성

| 컬럼명 | 설명 |
|---|---|
| 품목 | 카테고리별 식자재 품목 |
| 단위 | 품목별 사용 단위 |
| 기준 사용량 | 기준이 되는 사용량 입력값 (매번 직접 입력) |
| 실제 사용량 | 실제 사용량 입력값 |
| 차이(실제 - 기준) | 실제 사용량에서 기준 사용량을 뺀 값 (자동 계산) |
| 판정 | `정상`/`과다`/`부족` 중 라디오 버튼으로 하나 선택 (필수) |
| 원인 | 판정에 대한 원인을 적는 자유 텍스트 (선택, 미입력 가능) |
| 재고 | 현재 보유 재고 (직접 입력) |
| 전회차 단가 | 직전 기록의 공급단가 (DB에서 자동 조회, [7.2](#72-자동-계산-및-표시-규칙) 참고) |
| 공급단가 | 현재 공급 단가 (직접 입력) |
| 단가 상태(상승/동일/하락) | 전회차 단가와 공급단가의 단순 대소 비교 결과. `상승`/`하락`/`동일` 텍스트로 표시한다. |

> 식자재 기록에도 **담당자명**을 입력한다. (작성 담당자 식별용, SRS §5)

### 7.2 자동 계산 및 표시 규칙

| 구분 | 규칙 |
|---|---|
| 사용량 차이 | `실제 사용량 - 기준 사용량`으로 자동 계산한다. |
| 전회차 단가 조회 | 해당 품목의 직전 기록 공급단가를 자동으로 가져온다. 기본은 **전날** 기록, 전날 기록이 없으면 그 이전 **가장 최근** 기록일의 공급단가를 사용한다. |
| 단가 상태 판정 | `전회차 단가`와 `공급단가`를 **숫자로 단순 대소 비교**한다. 오늘 > 전회차 → 상승, 같으면 동일, 작으면 하락. |
| 품목별 단가 표시 | 단가 컬럼에 상승 → 빨강 텍스트, 하락 → 파랑 텍스트, 동일 → 회색 텍스트로 표시한다. 전회차 단가가 없으면 `-`로 표시한다. |

### 7.2.1 단가 상승 집계 표시

식자재 체크리스트 표의 **오른쪽 상단**에 전회차 대비 단가가 상승한 품목 수를 집계해 텍스트로 표시한다.

| 구분 | 규칙 |
|---|---|
| 집계 기준 | 모든 품목에 대해 `오늘 공급단가 > 전회차 단가`인(=상승한) 품목 개수를 센다. |
| 표시 형식 | `전회차 대비 N개 상승` 형태의 텍스트로 표시한다. (점/원형 표시는 사용하지 않음) |
| 텍스트 색상 | N = 1 → 초록, N = 2~3 → 노랑, N ≥ 4 → 빨강 |

### 7.3 단위 구성

| 구분 | 단위 / 옵션 |
|---|---|
| 기본 단위 | Kg, g, 단, 봉, 개, 망, 박스, 구 |
| 세부 옵션이 있는 단위 | Kg, g, 구 |
| Kg 세부 옵션 | 1, 2.7, 3, 5, 7.5, 10, 11, 13, 18, 20 |
| g 세부 옵션 | 250, 300, 500 |
| 구 세부 옵션 | 15, 30 |

> 위 **세부 옵션 값은 초기 시드**이며, 설정 페이지(식자재 탭)에서 추가/수정/삭제할 수 있다([9.1](#91-식자재-탭) 참고). 단위(Kg/g/단…) 목록 자체는 고정이다.

**단위 입력 동작**
- 단위 선택 시 **Kg / g / 구** 를 선택하면, 오른쪽에 세부 옵션을 고르는 셀렉트박스가 추가로 나타난다.
- 그 외 단위(단, 봉, 개, 망, 박스)는 세부 옵션 없이 사용자가 수량을 직접 입력한다.
- 품목별 단위 자동 매칭은 하지 않는다. 사용자가 입력 시 직접 선택한다. (단, 설정 페이지에서 품목별 기본 단위를 디폴트로 지정해 둘 수 있다 — [9](#9-설정-페이지-품목카테고리-마스터-관리) 참고)

## 8. 공산품 체크리스트

### 8.1 컬럼 구성

| 컬럼명 | 설명 |
|---|---|
| 품목 | 공산품 품목 |
| 단위 | 품목별 재고 관리 단위. 신규 기본값은 품목 마스터의 기본 단위이며, 그날 선택값은 기록 행에 저장한다. |
| 입고 | 그날 입고량 (입고가 있는 날만 입력, 평소 빈칸). MVP2 통계의 공산품 사용량 산출에 사용 (사용량 = 전일 재고 + 당일 입고 − 당일 재고). |
| 재고 | 현재 보유 재고 (직접 입력) |
| 여유재고 | 품목별 고정 기준값(예: 항상 1박스 남김). 품목 마스터에서 관리하며 체크리스트에서는 읽기 전용으로 표시한다. |

> 컬럼 표시 순서: 품목 - 단위 - 입고 - 재고 - 여유재고.

> 공산품 기록에도 **담당자명**을 입력한다. (작성 담당자 식별용, SRS §5)
>
> **1주 평균 사용량**은 입력 컬럼이 아니다. 일자별 기록을 바탕으로 **MVP2 통계 페이지**에서 매주 월요일 전주 기준으로 계산해 보여준다. → [MVP2 계획](../plans/mvp2.md)

### 8.2 단위 구성

## 9. 설정 페이지 (품목/카테고리 마스터 관리)

별도 설정 페이지를 제공한다. 상단에 **식자재 / 공산품 탭바**를 두고 탭별로 해당 도메인의 마스터만 관리한다. (한 화면에 식자재·공산품을 모두 두면 드래그앤드랍 순서변경 시 복잡하므로 분리한다.)

### 9.1 식자재 탭

| 기능 | 설명 |
|---|---|
| 카테고리 CRUD | 카테고리 추가 / 이름 수정 / 삭제(soft delete) |
| 품목 CRUD | 품목 추가 / 이름 수정 / 삭제(soft delete), 소속 카테고리 지정 |
| 품목 디폴트값 | 품목별 기본 단위, 기본 단위 옵션, 기본 기준 사용량 디폴트를 미리 지정해 입력 편의를 높인다. |
| 단위 옵션 관리 | 세부 옵션이 있는 단위(Kg/g/구)의 **옵션 값을 추가/수정/삭제**한다. (예: Kg에 `25` 추가) 단위(Kg/g/단…) 목록 자체는 고정이며 옵션 값만 편집한다. |
| 순서 변경 | 카테고리·품목 순서를 **드래그앤드랍**으로 위아래로 옮긴다(`sort_order` 갱신). |

### 9.2 공산품 탭

| 기능 | 설명 |
|---|---|
| 품목 CRUD | 품목 추가 / 이름 수정 / 삭제(soft delete) |
| 품목 디폴트값 | 품목별 기본 단위와 여유재고 기준값을 지정한다. |
| 순서 변경 | 품목 순서를 **드래그앤드랍**으로 옮긴다(`sort_order` 갱신). |

### 9.3 공통

| 항목 | 정책 |
|---|---|
| 동시성 | 마스터 수정은 행 단위 `version` optimistic locking을 적용한다. 또한 설정 변경 시 도메인별 카탈로그 버전을 올려, 같은 시점에 체크리스트를 수정하던 사람과 겹치면 **설정 편집을 우선**하고 체크리스트 저장을 거부·새로고침시킨다. ([11. 저장 및 동시성 정책](#11-저장-및-동시성-정책) 참고) |
| 편집 제외 | 진입 비밀번호, 지점 목록, 운영 체크리스트 항목 텍스트는 설정 페이지 편집 대상이 아니다(코드/환경변수/시드 고정). |

**품목 추가/삭제가 과거 기록에 미치는 영향** — [10. 데이터 모델 원칙](#10-데이터-모델-원칙)을 따른다.
- 품목 **추가**: 마스터에 행을 추가한다. 과거 기록은 영향받지 않으며(그 품목 줄이 없을 뿐), 추가 시점 이후 입력 화면에만 노출된다.
- 품목 **삭제**: 실제 삭제 대신 `is_active = false`(soft delete)를 기본으로 한다. 신규 입력 화면에는 더 이상 노출하지 않되, 과거 기록의 입력값은 그대로 보존한다.

## 10. 데이터 모델 원칙

> 핵심: **품목은 테이블 컬럼이 아니라 행(row)으로 모델링한다.** 품목 추가/삭제가 컬럼 추가/삭제가 되면 과거 데이터가 깨지므로, 정규화된 행 기반 구조를 사용한다.

| 원칙 | 내용 |
|---|---|
| 행 기반 품목 모델 | 품목은 `품목 마스터` 테이블의 행으로 관리한다. 화면에서는 표(컬럼)처럼 보이지만, 저장 구조는 (기록 1건) ↔ (품목별 입력 행 N개) 관계이다. |
| 기록/상세 분리 | `체크리스트 기록`(지점·영업일·종류·version·담당자) 1건에 `상세 입력 행`이 N개 매달리는 구조로 저장한다. |
| 참조 모델(이름) | 상세 입력 행은 품목을 `item_id`로 참조만 한다. **품목명/카테고리명은 행에 복사하지 않고 마스터를 JOIN해 표시**한다. 설정 페이지에서 이름을 바꾸면 과거 기록 표시도 함께 바뀐다. |
| 입력값 동결 | 상세 입력 행에는 **그날 사용자가 입력/선택한 값**(단위·단위옵션·기준/실제 사용량·판정/원인·재고·단가)을 저장한다. 마스터의 디폴트값을 바꿔도 과거 행 값은 바뀌지 않는다(그날의 사실로 동결). |
| 품목 추가 | 마스터에 행 insert. 과거 기록 backfill 불필요. |
| 품목 삭제 | `is_active = false` soft delete를 사용한다. 참조 모델은 마스터 행이 살아 있어야 JOIN이 되므로 **하드 삭제는 하지 않는다.** (하드 삭제 시 과거 기록의 이름 표시가 깨지고 MVP2 통계 데이터도 유실됨) |
| 중복 기록 방지 | `UNIQUE(branch_id, business_date, checklist_type)` 제약을 사용한다. |
| 동시성 제어 | `version` 컬럼 기반 optimistic locking. ([11](#11-저장-및-동시성-정책) 참고) |

## 11. 저장 및 동시성 정책

| 항목 | 정책 |
|---|---|
| 저장 단위 | 체크리스트별(운영 12 + 식자재 + 공산품 = 14) 저장 버튼으로 각각 저장한다. |
| 빈 값 처리 | 숫자 컬럼은 0/소수점/음수/빈 값을 모두 허용한다. 단, 빈 값이 있는 상태로 저장 시도하면 **저장 전 1회 경고(알림)** 를 띄우고, 사용자가 확인하면 그대로 저장한다. |
| 저장 후 열람 | 저장된 (지점·영업일·체크리스트)는 다른 사용자가 열람할 때 저장된 값으로 표시된다. |
| 동시 편집 충돌(체크리스트) | optimistic locking을 사용한다. 불러온 시점의 `version`이 저장 시점에도 최신일 때만 저장을 허용한다. 그 사이 다른 사람이 저장해 version이 올라갔다면, **저장을 거부하고 "다른 사람이 먼저 저장했습니다. 새로고침 후 다시 시도하세요"** 안내를 표시한다. (무조건 덮어쓰기/자동 머지는 사용하지 않는다) |
| 설정 ↔ 설정 충돌 | 마스터(카테고리/품목)도 행마다 `version`을 두고 optimistic locking을 적용한다. 같은 행을 두 사람이 동시에 수정하면 늦은 쪽을 거부하고 새로고침을 안내한다. |
| 설정 ↔ 체크리스트 충돌 (설정 우선) | 설정 변경은 테이블 여러 행에 걸쳐 시간이 걸릴 수 있고, 과거 체크리스트를 수정하는 사람과 겹칠 수 있다. **우선순위는 설정 편집자에게 둔다.** 도메인(식자재/공산품)별 **카탈로그 버전**을 두고, 설정에서 품목/카테고리를 추가·삭제(soft delete)·순서변경·수정하면 카탈로그 버전을 +1 한다. 체크리스트는 진입 시점의 카탈로그 버전을 들고 있다가, 저장 시 버전이 달라졌으면(=그 사이 설정이 바뀌었으면) **체크리스트 저장을 거부하고 "품목 설정이 변경되었습니다. 새로고침 후 다시 시도하세요"** 안내를 표시한다. (설정 편집자는 영향받지 않는다. 설정→체크리스트 순서면 정상, 체크리스트가 설정보다 늦게 끝나면 거부) |

> 충돌 감지의 정확한 저장 방식(전용 메타 행 vs 마스터 `version` 집계 fingerprint)은 API 명세에서 확정한다.
